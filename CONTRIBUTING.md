# Contributing to Chrome Extension Guide

Thank you for your interest in contributing to the Chrome Extension Guide! This document outlines how you can contribute to this project.

## How to Contribute

### Reporting Bugs

If you find a bug or issue:

1. Check if there's already an existing issue
2. If not, create a new issue using the [Bug Report template](.github/ISSUE_TEMPLATE/bug_report.md)
3. Include detailed steps to reproduce, expected vs actual behavior, and your environment

### Suggesting Features

Have an idea for improvement? We welcome feature requests:

1. Use the [Feature Request template](.github/ISSUE_TEMPLATE/feature_request.md)
2. Explain the use case and why it would be beneficial
3. Provide any relevant examples or mockups

### Pull Requests

We love contributions! Here's how to submit a PR:

1. Fork the repository
2. Create a feature branch: `git checkout -b feature/your-feature-name`
3. Make your changes following our code style guidelines
4. Write clear commit messages
5. Push to your fork and submit a pull request
6. Fill out the PR template completely

## Code Style

### TypeScript

- Use TypeScript 5.0+ features including strict mode
- Enable `noImplicitAny`, `strictNullChecks`, and `strictFunctionTypes`
- Prefer interfaces over types for object shapes
- Use JSDoc comments for public APIs

### General Guidelines

- Follow existing code formatting (prettier/eslint)
- Keep functions small and focused
- Write meaningful variable and function names
- Comment complex logic
- Test your changes

### Documentation

- Use clear, concise language
- Include code examples where appropriate
- Update table of contents if adding new sections
- Keep documentation in sync with code changes

## PR Process

### Before Submitting

1. **Test your changes** - Ensure all code examples work
2. **Update documentation** - Related docs must reflect your changes
3. **Check formatting** - Run linting and formatting tools
4. **Review your PR** - Check for typos and clarity

### PR Requirements

- [ ] Code follows the style guide
- [ ] Documentation is updated
- [ ] All tests pass
- [ ] PR description explains the change clearly
- [ ] Links to related issues (if applicable)

### Review Process

1. Maintainers will review your PR
2. Address any feedback promptly
3. Once approved, your PR will be merged

## Issue Templates

We use GitHub issue templates to keep things organized:

- [Bug Report](.github/ISSUE_TEMPLATE/bug_report.md) - Report issues and errors
- [Feature Request](.github/ISSUE_TEMPLATE/feature_request.md) - Suggest new features
- [Documentation](.github/ISSUE_TEMPLATE/documentation.md) - Improve docs

Please use the appropriate template when creating issues.

## Getting Help

- Open an issue for bug reports or feature requests
